# 1. 背景与目标

你要做的作品集项目是一个“可上线”的企业知识库问答系统：用 RAG（检索增强生成）把企业文档接入大模型，支持引用溯源、权限控制、评测与监控，体现工程化落地能力。

## 1.1 项目定位

- **定位**：面向企业内部员工的知识库问答（也可类比客服知识库）。
- **核心卖点**：回答可追溯（引用来源）、可治理（权限/审计/拒答）、可迭代（评测回归）、可运营（监控/成本/延迟）。

## 1.2 目标（成功标准）

- 能对“被收录的文档内容”进行准确回答，并提供**可点击引用**（至少 2 条引用片段）。
- 对“库内无证据/证据不足/权限不足”的问题，能**拒答并解释原因**，必要时引导用户补充信息或走“转人工/提工单”（可用模拟接口）。
- 支持**按用户角色/部门**限制文档可见性，检索与引用必须严格遵守权限。
- 具备**离线评测与回归门禁**：每次调整提示词/切分/检索配置后可一键跑评测并产出报告。
- 具备**基础可观测性**：延迟、错误率、token 消耗、检索命中率、拒答率可监控。

## 1.3 非目标（V1 不做）

- 不做通用 Agent 多工具编排（这是另一个作品集）。
- 不做复杂的多租户计费/订阅。
- 不做文档在线协作编辑、流程审批。
- 不做花哨的 UI 页面体系（V1 只保留必要页面/入口）。

# 2. 术语

- **文档（Document）**：企业知识源文件（PDF/HTML/Markdown/Docx 等）。
- **切分（Chunk）**：将文档拆成用于检索的段落/片段。
- **Embedding**：将文本映射到向量空间的表示。
- **向量库（Vector Store）**：存储向量与元数据并支持相似度检索。
- **混合检索（Hybrid Search）**：关键词检索 + 向量检索组合。
- **重排（Rerank）**：对初检结果按相关性再排序。

# 3. 用户与使用场景

## 3.1 用户角色

- **普通用户（Employee）**：提问、追问、查看引用、反馈回答是否有用。
- **管理员（Admin）**：导入文档、配置权限/元数据、触发重建索引、查看日志与评测报告。

（可选）

- **审计/运维（Ops）**：只读查看监控与审计日志。

## 3.2 核心场景（必须覆盖）

1. 员工问制度/流程：例如“报销的额度与发票要求是什么？”
2. 员工问产品/技术文档：例如“XX 接口的鉴权方式是什么？”
3. 权限隔离：A 部门用户提问 B 部门私密文档内容，系统必须拒答或只基于可见文档回答。
4. 无法回答：库内缺失/证据不足时，明确拒答并提示如何补充（上传文档/给关键词/走工单）。

# 4. 核心功能点（学习成果映射）

> 这一节把学习计划里的关键能力映射成系统需求，后续生成项目时以此为“必做清单”。

## 4.1 RAG 全链路

- 文档接入：导入→清洗→切分→向量化→入库。
- 检索策略：top-k、（可选）混合检索、（可选）重排。
- 生成策略：只基于检索证据回答；必须输出引用；证据不足拒答。

## 4.2 工程化与可靠性

- API 服务：FastAPI；统一错误码；健康检查；流式输出（SSE）。
- 超时/重试/降级：LLM 调用失败、向量库不可用时的可解释降级。
- 缓存：embedding 缓存、检索缓存、回答缓存（按 query+权限+索引版本）。

## 4.3 治理（权限/审计/安全）

- 权限：文档/Chunk 元数据带权限位；检索端强制过滤。
- 审计：记录每次问答的检索结果、引用、模型参数、token 消耗。
- 安全：提示词注入基础防护；敏感信息脱敏（最小可用）。

## 4.4 评测与质量闭环

- 回归集：50–100 条问题，含期望行为（答对/拒答/必须引用/权限拒答）。
- 指标：正确率、引用率、拒答率、幻觉率（或“无证据回答率”）、延迟、成本。
- 评测报告：按错误类型分类（检索失败/切分不佳/生成不遵循引用/权限泄露等）。

# 5. 产品范围与页面（V1）

## 5.1 页面/入口

- **问答页（必需）**
    - 输入框 + 历史对话
    - 流式输出
    - “引用来源”区域（可展开查看片段与来源链接/文档名）
    - “有用/无用”反馈

- **管理入口（最小化，二选一即可）**
    - A：一个管理页（导入文档、查看索引状态、触发重建、配置权限）
    - B：不做管理页，改为管理 API + 简单脚本/CLI（更利于快速交付）

> 建议 V1 选 B：用脚本/命令行导入，减少前端页面数量，把精力集中在 RAG 质量与治理。

# 6. 功能需求（Functional Requirements）

## 6.1 文档导入与索引构建

### 6.1.1 支持的数据源（V1 最小集合）

- 必需：Markdown（.md）
- 可选：HTML（.html）或 PDF（二选一）

### 6.1.2 导入流程

- 管理员可通过脚本或 API 导入文件夹/单文件。
- 系统为每个文档生成并保存：
    - doc_id、标题、来源路径/URL、更新时间、所属部门/权限、哈希（用于增量更新）
- 文档解析失败要可追踪：记录失败原因并允许重试。

### 6.1.3 清洗与切分

- 清洗：去除无意义导航/页眉页脚（PDF/HTML 时尤其重要）；保留结构信息（标题层级/列表）。
- 切分策略：
    - chunk_size（字数/Token）与 overlap 可配置
    - chunk 必须携带：doc_id、章节标题（若有）、页码/段落序号、权限信息、来源定位信息

### 6.1.4 向量化与入库

- 对每个 chunk 计算 embedding，写入向量库。
- embedding 需缓存（按 chunk_hash），避免重复计算。
- 支持“增量更新”：文档内容未变化则跳过重建；变化则只更新受影响 chunks。

## 6.2 检索

- 输入：query + user_identity（含角色/部门）
- 输出：候选 chunks（含分数、来源元数据）

### 6.2.1 权限过滤（强约束）

- 检索必须先过滤不可见文档/chunk，再做排序。
- 任何情况下都不允许把不可见 chunk 作为引用返回。

### 6.2.2 检索策略（V1）

- 向量检索 top-k（k 可配置）
- （可选）混合检索：关键词 BM25/简单倒排 + 向量检索合并
- （可选）重排：对 top-k 候选进行 rerank，提高相关性

### 6.2.3 失败兜底

- 若检索不到足够证据（例如低于阈值或 top-k 为空），进入拒答策略（见 6.3）。

## 6.3 生成（回答与引用）

### 6.3.1 回答规范

- 回答必须基于检索证据；不允许无证据编造。
- 回答必须包含：
    - 简明结论（1–3 句）
    - 关键依据点（可列表）
    - 引用列表（至少 2 条，若证据不足则拒答）

### 6.3.2 引用溯源

- 每条引用至少包含：文档名、章节/页码（若有）、chunk 片段摘要、来源链接/路径。
- 前端可以点击展开查看引用片段全文（或片段）。

### 6.3.3 拒答策略（必需）

触发条件（任一满足即可）：

- 检索证据不足（低相关/空结果）
- 用户权限不足（仅检索到不可见文档）
- 问题超出知识库范围（开放式创作、与业务无关等）

拒答输出必须包含：

- 明确说明“无法基于当前知识库回答”的原因（证据不足/权限不足/缺少信息）
- 给出下一步建议：
    - 建议补充关键词/时间范围/系统名
    - 或提示“可提交工单/联系管理员补充文档”（可提供模拟入口）

## 6.4 对话与流式体验

- 后端支持 SSE 流式输出（token/句子级增量）。
- 前端支持：
    - 实时增量渲染
    - 中断生成
    - 失败重试（同一轮问题可重试，记录重试次数）

## 6.5 反馈与日志

- 用户可对每次回答做“有用/无用”反馈（可选带原因标签）。
- 系统记录以下审计信息（最小集合）：
    - user_id、query、时间
    - 检索到的 doc_id/chunk_id 列表与分数（允许脱敏/只存 id）
    - 实际引用的 chunks
    - 模型与参数（温度等）
    - token 消耗、耗时、是否拒答、错误码

## 6.6 评测与回归

- 提供离线评测脚本/命令（例如 `python -m eval.run` 的形式），输入：回归集 + 当前配置版本。
- 评测输出：
    - 汇总指标（正确率/引用率/拒答率/越权率/延迟/成本）
    - 失败用例清单（含检索结果与引用）
    - 错误分类统计（检索失败、生成不遵循格式、引用不准确、权限问题等）
- 回归门禁建议：
    - 越权率必须为 0
    - 引用率 ≥ 95%（能回答的样本中）
    - 拒答准确率（该拒答的拒答）≥ 90%（可先用人工抽检）

# 7. 非功能需求（Non-Functional Requirements）

## 7.1 性能与成本

- 目标 P95 延迟（单轮问答）：
    - V1 本地/单机可先不硬卡，但需要记录并可优化
- 成本：记录每次请求 token；支持按天汇总。
- 并发：至少支持 10 并发请求不崩溃（单机）。

## 7.2 可用性与可靠性

- 提供 `/health` 健康检查。
- LLM 调用失败要返回可解释错误，并可降级到“只返回检索结果片段”（可选）。

## 7.3 安全与合规（最小可用）

- 基础提示词注入防护：
    - 系统提示明确“只能使用提供的上下文”
    - 对用户输入做简单规则检测（例如要求泄露系统提示/让你忽略规则等）并触发更保守策略
- 敏感信息：
    - 对日志中的 query 与引用片段支持脱敏（可配置开关）

## 7.4 可观测性

- 结构化日志（JSON）：request_id、user_id、耗时、token、错误码。
- 指标（最小集合）：请求量、P50/P95 延迟、错误率、拒答率、命中率（检索有结果占比）、平均引用数、token 消耗。

# 8. 数据模型（建议）

V1 可用 SQLite/Postgres 保存业务元数据（文档、chunk 元信息、反馈、日志索引），向量库单独存储向量。

- Document：id、title、source、updated_at、owner_dept、visibility（role/dept）、hash
- Chunk：id、doc_id、text、section、page/idx、chunk_hash、visibility
- QueryLog：id、user_id、query、retrieved_chunk_ids、cited_chunk_ids、is_refusal、latency_ms、token_in/out、created_at
- Feedback：id、query_log_id、vote、reason(optional)
- EvalCase：id、question、expected_behavior（answer/refusal/permission_refusal）、gold_doc(optional)

# 9. 技术栈建议（贴合学习计划）

> 这里只给出建议选型，后续生成项目时可根据你已有经验微调。

- 后端：Python + FastAPI
- LLM：任一可用的 Chat Completions API（支持流式与结构化输出更好）
- Embedding：与所选模型/服务匹配的 embedding API
- 向量库：Chroma
- 元数据存储：SQLite（V1 最快）
- 缓存（可选）：Redis
- 前端：Vue3 + TS + Vite；SSE 流式渲染
- 部署：Docker / Docker Compose

# 10. 里程碑（与学习周次对齐）

- M1（RAG 闭环）：导入 Markdown → 切分 → 向量库 → top-k 检索 → 返回片段（可用脚本演示）
- M2（问答 V1）：流式对话 + 回答带引用 + 拒答策略
- M3（治理）：权限过滤 + 审计日志 + 基础防注入
- M4（评测）：回归集 + 一键评测 + 报告 + 回归门禁
- M5（上线能力）：缓存/限流（最小可用）+ 指标监控

# 11. 验收清单（交付时自测）

- 引用：能回答的问题，引用条数 ≥ 2，且点击可定位到来源文档/片段。
- 拒答：无证据/权限不足时必须拒答，且给出下一步建议。
- 权限：不同角色/部门用户对同一问题的结果不同；越权引用为 0。
- 评测：能一键跑回归集并生成报告；失败用例可定位到“检索/切分/生成/权限”的原因。
- 监控：能看到延迟、错误率、token 消耗、拒答率、命中率的统计。

# 12. 约束与取舍（写在前面，避免做大）

- V1 优先保证：**权限正确性 > 引用溯源 > 拒答可解释性 > 检索效果 > 体验细节**。
- 数据源优先从你手头可获得的公司/公开文档开始；先把评测集做起来再谈“效果优化”。
